### Sprint 25 — Frontend Resilience & Observability (Phase A — backend health & CSP support)

> [docs/roadmap.md § Sprint 25](docs/roadmap.md). Phase A unblocks the UI repo's remaining slices: B5 (CSP report-only header in `staticwebapp.config.json`) needs A3 to ship first, and B3/B4 (deploy-time + post-deploy smokes) need a healthy api with the new `/health/live` contract.

- **A1 — `/health/live` SPA-polling contract** ([src/tagpulse/api/routes/health.py](src/tagpulse/api/routes/health.py), [Dockerfile](Dockerfile), [.github/workflows/build-and-push.yml](.github/workflows/build-and-push.yml)). Body shape switches from the legacy `{"status":"ok"}` to `{"status":"alive","version":"<git-sha>","build_time":"<iso8601>"}`; `Cache-Control: no-store` is set on the response so the SWA edge / browser back-cache never memoize a stale "alive" (which would silently defeat the SPA's startup gate when the api goes down). Build identity is baked at image build time via the new `BUILD_VERSION` + `BUILD_TIME` Docker build-args; the GHA matrix populates them with `${{github.sha}}` and a UTC ISO-8601 timestamp computed once per job. Local `docker build` without `--build-arg` keeps the dev sentinels (`dev` / `unknown`). `/health` (no path suffix) is unchanged for k8s probes that pre-date `/live`.
- **A2 — CORS preflight max-age** ([src/tagpulse/core/config.py](src/tagpulse/core/config.py), [src/tagpulse/api/main.py](src/tagpulse/api/main.py)). New `Settings.cors_preflight_max_age_seconds` (default 0; the strict-mode validator forces 600 in non-dev when left at 0, and explicit non-zero values from `CORS_PREFLIGHT_MAX_AGE_SECONDS` survive untouched). Wired into `CORSMiddleware`'s `max_age=`. Cuts 60–80ms off the first-paint-to-login-button latency on cold tabs by letting the browser cache the OPTIONS preflight result.
- **A3 — `POST /security/csp-report` endpoint** ([src/tagpulse/api/routes/security.py](src/tagpulse/api/routes/security.py)). Accepts both browser report shapes: the Chromium/Safari legacy `application/csp-report` envelope (`{"csp-report": {...}}`) and the Reporting API `application/reports+json` array of `{type, body}` envelopes. Emits a structured WARN log with `(client_ip, blocked_uri, document_uri, violated_directive, source_file, line_number, column_number, user_agent)` and increments the Prometheus counter `tagpulse_csp_violations_total{directive=...}`. Per-IP sliding-window rate limit at 10 reports / 60s — same in-process trade-off as the Sprint 22 A4 limiter; a multi-replica api may admit up to 10·N reports/min cluster-wide, which is fine for this endpoint. Open without auth (browsers don't send credentials on report POSTs by design); `/security/csp-report` is added to the tenant-keyed limiter's bypass list. Returns `204` on success, `429` when capped, and `204` (logged + dropped) on malformed JSON so a misbehaving extension cannot 500 the api.
- **A4 — Runbook update** ([docs/runbooks/azure-first-deploy.md](docs/runbooks/azure-first-deploy.md)). `/health/live` smoke now asserts the new `{status, version, build_time}` shape + `Cache-Control: no-store`. New **§3b — SPA-vs-api consistency smoke** pulls `index.html` from the SWA, asserts `/health/ready` against the embedded api URL, and verifies the SWA hostname is in `config.cors.allow_origins` — catches the most insidious post-deploy failure (SPA built against a stale `VITE_API_BASE_URL`). New **§3c — CSP violation triage** documents the report shape, the Log Analytics KQL for top violations, and the triage priority order (`script-src` / `connect-src` are policy actually breaking the SPA; `img-src` / `font-src` are typically third-party extension noise).
- **Tests** — 13 new unit tests in [tests/unit/test_sprint25_phase_a.py](tests/unit/test_sprint25_phase_a.py) cover the `/health/live` contract (body shape, `Cache-Control: no-store`, latency budget, legacy `/health` unchanged), the CORS max-age policy in dev / non-dev / explicit-override paths, and the CSP-report endpoint across both report shapes, the per-IP rate limit, the no-auth contract, and the malformed-JSON path. `make check` clean (516 unit tests + 1 skipped, ruff + mypy green).

### Hotfix — Azure MQTT broker reachability (`/health/ready` 503)

- **Emit `MQTT_BROKER_HOST` + `MQTT_BROKER_PORT` from Bicep** ([deploy/azure/bicep/modules/container-app.bicep](deploy/azure/bicep/modules/container-app.bicep)). Pydantic `Settings` reads `MQTT_BROKER_HOST` / `MQTT_BROKER_PORT`, **not** `MQTT_BROKER_URL`; the Bicep contract was URL-only, so on Azure both the `/health/ready` TCP probe and the worker subscriber fell back to the Settings defaults (`localhost:1883`) and the readiness check stayed 503 with `mqtt: down (Connection refused)` even though the broker container was reachable from the Container Apps subnet. The module now derives `brokerHost` / `brokerPort` by stripping `mqtt://` / `mqtts://` from `mqttBrokerUrl` and injects both env vars into the api + worker container env (alongside the existing `MQTT_BROKER_URL` for forward-compat).
- **Mosquitto entrypoint chowns the generated password file** ([docker/mosquitto-entrypoint.sh](docker/mosquitto-entrypoint.sh)). When the entrypoint runs as root and writes `/mosquitto/config/mosquitto.passwd` via `mosquitto_passwd -b -c`, the file is owned by root and the unprivileged `mosquitto` user (uid 1883) cannot read it — broker exits with `Error: Unable to open password file ... Permission denied`. The script now `chown mosquitto:mosquitto` + `chmod 0640` the file after generation so the broker can start under its own uid.

### Hotfix — Azure PG Flex compatibility for TimescaleDB

- **Pin `postgresVersion` to `15`** ([deploy/azure/bicep/modules/postgres.bicep](deploy/azure/bicep/modules/postgres.bicep)). Azure Database for PostgreSQL Flexible Server **does not** ship the TimescaleDB extension on PG16 yet; `CREATE EXTENSION timescaledb` on PG16 silently drops the connection mid-statement. The Bicep default was `'16'`, so a fresh provision produced an unusable database. Pinning to PG15 (which is fully supported) is the only reliable path until Microsoft adds PG16 support.
- **Include platform `shared_preload_libraries` defaults when adding `timescaledb`** ([deploy/azure/bicep/modules/postgres.bicep](deploy/azure/bicep/modules/postgres.bicep)). The Flex Server `shared_preload_libraries` parameter **overwrites** the platform value rather than appending; setting it to `'timescaledb'` alone removes the platform-managed `pg_cron` and `pg_stat_statements` and leaves the server in a state where `CREATE EXTENSION timescaledb` again drops the connection. Value is now `'pg_cron,pg_stat_statements,timescaledb'` and the parameter has a long-form comment explaining the overwrite-not-append behavior.
- **Migration 031 is license-aware and idempotent** ([migrations/versions/031_telemetry_subject_kinds_and_caggs.py](migrations/versions/031_telemetry_subject_kinds_and_caggs.py)). Azure PG Flex ships TimescaleDB in **Apache-2** edition only; continuous aggregates (`CREATE MATERIALIZED VIEW … WITH (timescaledb.continuous)` + `add_continuous_aggregate_policy`) are TSL-licensed and fail with `FeatureNotSupportedError: functionality not supported under the current "apache" license`. The `tenants.telemetry_subject_kinds` column is now added with `ADD COLUMN IF NOT EXISTS` so partial runs don't break re-execution; `current_setting('timescaledb.license', true) = 'apache'` short-circuits both `upgrade()` and `downgrade()` past the cagg DDL. Local Docker (TSL edition) gets caggs as before; Azure Flex skips them and the application falls through to live `time_bucket()` queries.
- **`TimescaleTelemetryReadingsRepository.aggregate()` falls back to live `time_bucket` when caggs are absent** ([src/tagpulse/repositories/timescaledb/telemetry.py](src/tagpulse/repositories/timescaledb/telemetry.py)). A process-cached `pg_class` probe (`_caggs_available`) detects whether `cagg_telemetry_1m` and `cagg_telemetry_1h` exist; when they don't, both the `bucket_seconds=60` and `=3600` paths route to the same live `time_bucket(make_interval(secs => :secs), …)` SQL the `else` branch already used. Apache-edition deployments serve `/telemetry/aggregates` correctly with no client-visible difference (the only cost is computing aggregates from the raw hypertable on each request).

### Hotfix — Azure deploy DATABASE_URL incompatible with asyncpg

- **Bicep `DATABASE_URL` switched from `?sslmode=require` to `?ssl=require`** ([deploy/azure/bicep/modules/container-app.bicep](deploy/azure/bicep/modules/container-app.bicep), [deploy/azure/bicep/modules/migrations-job.bicep](deploy/azure/bicep/modules/migrations-job.bicep)). Migrations job + api + worker all use `postgresql+asyncpg://...`; asyncpg's `connect()` does not accept the libpq query parameter `sslmode`, so the migrations job crashed with `TypeError: connect() got an unexpected keyword argument 'sslmode'` and the api revision never became ready (the readiness probe stayed 503 because the schema was never applied). Symptom: `curl https://${apiFqdn}/health/live` hangs because both revisions of `tpdev-api` are `ActivationFailed` with 0/1 replicas ready. Diagnosis path: Container Apps env Log Analytics → `ContainerAppConsoleLogs_CL | where ContainerName_s == 'migrations'` → traceback shows the offending kwarg. Fix is a one-character change (`sslmode` → `ssl`); both forms mean "TLS required" but only the latter is asyncpg-native. ARM template `deploy/azure/bicep/main.json` regenerated.

### `azd up` postdeploy hook robustness

- **Drop `az deployment sub show --name $AZURE_ENV_NAME` lookups** ([azure.yaml](azure.yaml)).
