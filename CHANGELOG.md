- **Sprint 35 Phase C — Labels deep-object filter (ADR 020).** Lights up the `?labels[KEY]=V1,V2` filter on the four entity list endpoints (`GET /assets`, `GET /sites`, `GET /zones`, `GET /devices`) that Phase B intentionally deferred. New module `src/tagpulse/api/label_filter.py` exposes two purpose-built primitives. (1) `parse_label_filter(query_params)` — pure parser over Starlette's `QueryParams.multi_items()` so duplicate `labels[KEY]=…` entries on the same request merge correctly; returns `dict[str, list[str]] | None` (`None` when no `labels[...]` keys are present so the call site can short-circuit), raises `LabelFilterError(ValueError)` on any guard-rail violation. Guard rails mirror ADR 020 §"Filter encoding" verbatim: ≤ 5 distinct keys/request (`LABEL_FILTER_MAX_KEYS`), ≤ 20 values/key (`LABEL_FILTER_MAX_VALUES_PER_KEY`), key regex `^[A-Za-z0-9_.+$]{3,24}$` and value regex `^[A-Za-z0-9._-]{1,64}$` (imported from `schemas.py` so the catalog and the filter can't drift), and any empty segment — leading comma, trailing comma, double comma, or fully-empty `labels[k]=` — rejects with `"Empty value …"` rather than silently matching nothing. Keys fold to lowercase so `labels[Location]=` and `labels[location]=` merge into one bucket (matches the catalog's `(tenant_id, entity_type, lower(key))` functional unique index from migration 039); values stay case-sensitive. (2) `apply_label_filter(stmt, *, tenant_id, entity_type, entity_id_col, labels)` — pure SQLAlchemy applicator that appends one correlated `EXISTS` per key onto the caller's `Select`, joining `entity_labels → labels` and binding `LabelModel.tenant_id = :tid AND LabelModel.entity_type = :et AND lower(LabelModel.key) = lower(:key) AND EntityLabelModel.value IN (:v1, :v2, …)`. Tenant scoping is enforced inside the EXISTS — a guess-the-UUID attack on another tenant's label can't bypass the `tenant_id` check on the join side. The four list routes now accept `request: Request`, run the parser, surface `LabelFilterError` as a 400, and pass `labels=` through their service. Service signatures (`AssetService.list_assets`, `SiteZoneService.list_sites` / `.list_zones`, `DeviceService.list_devices`) gained the `labels: dict[str, list[str]] | None = None` kwarg (default `None` preserves existing call sites). Repository `.list()` methods on `TimescaleAssetRepository`, the site + zone repos, and `TimescaleDeviceRepository` now invoke `apply_label_filter` between their existing per-field WHEREs and the `ORDER BY / LIMIT / OFFSET` tail. `DeviceRepository` Protocol updated to mirror the new kwarg. **Why the manual `Request.query_params` parse instead of a typed FastAPI `Query` parameter:** OpenAPI 3.0 supports `style: deepObject, explode: true` (which is what ADR 020 specifies) but FastAPI's `Query()` machinery doesn't surface deep-object query params as a single typed value — the `[`/`]` characters in the param name aren't legal Python identifiers and Pydantic v2 won't bind them. Going through `request.query_params.multi_items()` keeps the route signature clean, lets the guard-rail errors return a real 400 with a useful message, and crucially **does not change the generated `openapi.json`** (the spec gate stays green without manual schema patches — verified locally). **AND across keys, OR within values** is the SQL contract: each key adds one `EXISTS`, and within an EXISTS the value list compiles to `entity_labels.value IN (…)` — equivalent to the ADR's example SQL. Performance shape is index-friendly: each `EXISTS` is a fast probe via `entity_labels(label_id, entity_id)` PK + `labels(tenant_id, entity_type, lower(key))` functional unique index. **Tests:** 50 new unit tests in `tests/unit/test_label_filter.py` cover parser happy-path (single pair, multi-key AND, comma-split OR, the four-key combined form from the ADR), duplicate-key merge, dedup-within-key, key case-folding, the full guard-rail matrix (max keys + at-max accepted, max values + at-max accepted, parametrised key + value regex with allowed and disallowed inputs, empty/leading/trailing/double comma all rejected), `apply_label_filter` pass-through for `None` and `{}`, and SQL-shape assertions (compile against the postgres dialect with `literal_binds=True` so the test reads what the DB actually sees: one EXISTS per key, `entity_type` bound to the singular form, key bound lowercased). The existing fake repositories in `tests/unit/test_device_service.py` and `tests/unit/test_sites_zones_service.py` picked up the new `labels=` kwarg (accepted-but-ignored at the fake layer — repository-level SQL behaviour is covered in the new test module). Full unit suite now 807 tests (+50 from Phase B's 757). `src/tagpulse/api/label_filter.py`, `src/tagpulse/api/routes/assets.py`, `src/tagpulse/api/routes/devices.py`, `src/tagpulse/api/routes/sites_zones.py`, `src/tagpulse/api/services/asset_service.py`, `src/tagpulse/api/services/device_service.py`, `src/tagpulse/api/services/sites_zones_service.py`, `src/tagpulse/repositories/protocols.py`, `src/tagpulse/repositories/timescaledb/assets.py`, `src/tagpulse/repositories/timescaledb/devices.py`, `src/tagpulse/repositories/timescaledb/sites_zones.py`, `tests/unit/test_label_filter.py`, `tests/unit/test_device_service.py`, `tests/unit/test_sites_zones_service.py`.

- **Sprint 35 Phase A — Labels catalog schema (ADR 020).** Migration `039_labels_catalog.py` introduces two tenant-scoped tables. (1) `labels` — catalog of label *keys* scoped to one `entity_type` (`asset` / `site` / `zone` / `device` / `category`); functional UNIQUE index on `(tenant_id, entity_type, lower(key))` enforces case-insensitive uniqueness per entity-type slot (so `location` on Sites and `location` on Assets are independent). DB-side CHECK constraints enforce the `entity_type` enum, the key regex `^[A-Za-z0-9_.+$]{3,24}$`, and the optional `color` regex `^#[0-9A-Fa-f]{6}$`. (2) `entity_labels` — polymorphic association `(label_id, entity_id, value)`; `label_id` FK is **`ON DELETE RESTRICT`** so deleting a catalog row that's still referenced returns 409 + `association_count` at the API layer (matches the Categories pattern from ADR 019, *not* the CASCADE the original ADR draft showed). `entity_id` has no FK — it points polymorphically at `assets` / `sites` / `zones` / `devices` / `categories` rows; orphan cleanup happens in the entity-delete handlers in Phase B, not in the DB. Value regex is `^[A-Za-z0-9._-]{1,64}$` (hyphen added vs the ADR draft so `warehouse-a` style values actually validate; matches k8s/AWS/GCP tag value conventions). A `enforce_label_cap()` PL/pgSQL function + `BEFORE INSERT` trigger backstops the 30-labels-per-entity cap; SQLSTATE `23514` surfaces as a 409 in the API. RLS is enabled on both tables (theatre-only per the repo convention; the app filters explicitly by `tenant_id` in every query). `LabelModel` + `EntityLabelModel` ORM classes in `src/tagpulse/models/database.py` mirror the migration; `created_by` / `updated_by` are plain `UUID NULL` columns without an FK (no `users` table in TagPulse — auth is JWT-based, mirrors `audit_logs.user_id` from migration 015). 16 new unit tests in `tests/unit/test_labels_models.py` introspect the ORM shape so anyone editing the migration without touching the models (or vice versa) trips a fast unit test before the integration suite runs. Three contradictions in the merged ADR 020 are fixed alongside the schema that crystallizes them: (a) `ON DELETE CASCADE` → `ON DELETE RESTRICT` (consistent with the 409 spec in the §"API path deviation" section), (b) value regex `^[A-Za-z0-9_.]{1,64}$` → `^[A-Za-z0-9._-]{1,64}$` (so the §"Filter encoding" `warehouse-a` examples validate), (c) `created_by/updated_by UUID REFERENCES users(id)` → `UUID` (no FK; no `users` table exists). The original `/v1/tenants/{slug}/...` API surface block in the ADR is now explicitly labelled "superseded by §'API path deviation' below" so future readers don't think two endpoint shapes were specified. `migrations/versions/039_labels_catalog.py`, `src/tagpulse/models/database.py`, `tests/unit/test_labels_models.py`, `docs/adr/020-labels-first-class.md`.

### Ops — `azd-job.sh` log-streaming fix (kills the `azd-kv-get.sh` flake)

- **Symptom.** `scripts/azd-kv-get.sh dev <secret>` (and any `azd-job.sh`-based wrapper) routinely returned `error: could not extract output from job log` for short-lived scripts (`get_kv_secret.py`, ~2s wall-clock). The wrapper waited 20 s then issued exactly **one** Log Analytics query — but Container Apps → Log Analytics ingestion lag is highly variable in centralus (typically 60–120 s, sometimes >2 min), so the query landed before the row was indexed and returned empty. Operators had to re-run with `--update-only` after a manual `sleep 90`. Made `azd-kv-get.sh` effectively unusable for interactive secret retrieval.
- **Fix — two-tier log fetch in [scripts/azd-job.sh](scripts/azd-job.sh).**
  1. **Wait for terminal status first** (5 s poll instead of 10 s) so we don't race a still-running container.
  2. **Try the Container Apps data-plane log endpoint first** — `az containerapp job logs show --execution <name> --container tools --format text --tail 300 --only-show-errors`. This hits the live-logs API on the ACA env and bypasses Log Analytics entirely; ~instant when it works. Strip the `<RFC3339> <stream> F ` prefix the data-plane format adds so callers get bare log lines (matters for `azd-kv-get.sh`, which greps between sentinels and pipes the value to `export FOO=$(…)`).
  3. **Fall back to Log Analytics with retry** — 12× 20 s poll (4-min cap) when the data-plane returns nothing (e.g. replica already garbage-collected on `--update-only`). Eliminates the single-shot race.
  - Net: `scripts/azd-kv-get.sh dev --list` and `scripts/azd-kv-get.sh dev <name>` now return clean output in <30 s for the live path, with a graceful fallback for stale executions.
- **`scripts/azd-kv-get.sh` — strip the wrapper's 4-space indent inside the sentinel block** ([scripts/azd-kv-get.sh](scripts/azd-kv-get.sh)) so `printf '%s\n' "$(echo "$PAYLOAD" | tail -1)"` returns the raw secret value, not `    <value>` with leading whitespace. Header comment updated to reflect the data-plane-first strategy.
- **End-to-end validated** against `tpdev`: `--list` returns the 6 KV secrets in <20 s; single-secret retrieval (`mqtt-broker-password`, 32-byte value) round-trips cleanly into a shell variable.
- **Backlog (Sprint 24+).** "Production-grade Bastion + jumpbox standard for staging + production envs" already on the [Sprint 23 deferred list](docs/roadmap.md#deferred-to-sprint-24) covers the structural alternative — laptop direct-to-KV access via a VNet-resident jumpbox — for operators who prefer Azure Portal "Show Secret Value" over the tools-job round-trip.

### Hotfix — Azure PG Flex compatibility for TimescaleDB

- **Pin `postgresVersion` to `15`** ([deploy/azure/bicep/modules/postgres.bicep](deploy/azure/bicep/modules/postgres.bicep)). Azure Database for PostgreSQL Flexible Server **does not** ship the TimescaleDB extension on PG16 yet; `CREATE EXTENSION timescaledb` on PG16 silently drops the connection mid-statement. The Bicep default was `'16'`, so a fresh provision produced an unusable database. Pinning to PG15 (which is fully supported) is the only reliable path until Microsoft adds PG16 support.
- **Include platform `shared_preload_libraries` defaults when adding `timescaledb`** ([deploy/azure/bicep/modules/postgres.bicep](deploy/azure/bicep/modules/postgres.bicep)). The Flex Server `shared_preload_libraries` parameter **overwrites** the platform value rather than appending; setting it to `'timescaledb'` alone removes the platform-managed `pg_cron` and `pg_stat_statements` and leaves the server in a state where `CREATE EXTENSION timescaledb` again drops the connection. Value is now `'pg_cron,pg_stat_statements,timescaledb'` and the parameter has a long-form comment explaining the overwrite-not-append behavior.
- **Migration 031 is license-aware and idempotent** ([migrations/versions/031_telemetry_subject_kinds_and_caggs.py](migrations/versions/031_telemetry_subject_kinds_and_caggs.py)). Azure PG Flex ships TimescaleDB in **Apache-2** edition only; continuous aggregates (`CREATE MATERIALIZED VIEW … WITH (timescaledb.continuous)` + `add_continuous_aggregate_policy`) are TSL-licensed and fail with `FeatureNotSupportedError: functionality not supported under the current "apache" license`. The `tenants.telemetry_subject_kinds` column is now added with `ADD COLUMN IF NOT EXISTS` so partial runs don't break re-execution; `current_setting('timescaledb.license', true) = 'apache'` short-circuits both `upgrade()` and `downgrade()` past the cagg DDL. Local Docker (TSL edition) gets caggs as before; Azure Flex skips them and the application falls through to live `time_bucket()` queries.
- **`TimescaleTelemetryReadingsRepository.aggregate()` falls back to live `time_bucket` when caggs are absent** ([src/tagpulse/repositories/timescaledb/telemetry.py](src/tagpulse/repositories/timescaledb/telemetry.py)). A process-cached `pg_class` probe (`_caggs_available`) detects whether `cagg_telemetry_1m` and `cagg_telemetry_1h` exist; when they don't, both the `bucket_seconds=60` and `=3600` paths route to the same live `time_bucket(make_interval(secs => :secs), …)` SQL the `else` branch already used. Apache-edition deployments serve `/telemetry/aggregates` correctly with no client-visible difference (the only cost is computing aggregates from the raw hypertable on each request).
